# Agent task: build a literary place index from ebooks and PDFs

You are building a Python command-line tool that creates a place index from an author’s works.

## Goal

Create a reproducible pipeline that reads all `.epub` and `.pdf` files in a local `data/` folder, extracts place mentions, normalizes them, and outputs CSV files containing a literary place index.

The output should help answer:

* Which places are mentioned?
* In which work?
* On which page/chapter/chunk?
* What city/region/country does the place belong to, when confidently known?

## Project context

This dataset is intended for later examination by experts on the author and for later transformation into one or more data visualizations.

Prioritize transparency, provenance, reproducibility, and easy manual review over aggressive automation. It is better to preserve uncertainty in the CSVs than to over-normalize or invent confident geographic metadata.

## Input

A project folder with this structure:

```text
project/
  data/
    work_1.epub
    work_2.pdf
    work_3.epub
  output/
  src/
```

The `data/` folder contains both PDFs and EPUBs.

## Required outputs

Create these files in `output/`:

```text
output/place_mentions.csv
output/place_index.csv
output/processing_log.csv
```

### `place_mentions.csv`

One row per mention.

Required columns:

```text
work
source_file
source_type
source_ref
chunk_id
place_text
normalized_place
place_type
city
region
country
confidence
context
```

Where:

* `work` = title inferred from filename unless better metadata is available
* `source_file` = original filename
* `source_type` = `pdf` or `epub`
* `source_ref` = page number for PDFs, chapter/section for EPUBs when available, otherwise chunk reference
* `chunk_id` = stable id
* `place_text` = exact text found in the work
* `normalized_place` = canonical place name
* `place_type` = city, street, district, landmark, building, country, region, river, other
* `city`, `region`, `country` = filled only when confident
* `confidence` = 0–1
* `context` = short surrounding sentence or paragraph

### `place_index.csv`

One row per normalized place.

Required columns:

```text
normalized_place
place_type
city
region
country
works
mention_count
sources
example_context
```

Where:

* `works` = semicolon-separated list of works
* `sources` = compact list of source refs
* `example_context` = one representative context

### `processing_log.csv`

One row per file/chunk with status.

Required columns:

```text
source_file
chunk_id
status
error
characters
mentions_found
```

## Implementation details

Create the following files:

```text
src/__init__.py
src/place_index.py
pyproject.toml
uv.lock
README.md
```

Use [`uv`](https://docs.astral.sh/uv/) for Python dependency management. Do not create or rely on `requirements.txt` for this project. The README should show `uv sync` for installation and `uv run ...` for execution.

Use `output/intermediate_mentions.jsonl` for checkpointing. Each processed chunk should be written immediately after processing so interrupted runs can resume safely.

When `--resume` is passed, skip chunks already present in `intermediate_mentions.jsonl` with status `success`. Failed chunks may be retried.

Use stable chunk IDs in this format:

```text
{source_file_stem}:{source_ref}:{chunk_number}
```

If `source_ref` contains characters unsuitable for IDs, sanitize it consistently. Chunk IDs must be deterministic across reruns with the same input files and chunking settings.

If metadata is unavailable, infer `work` from the filename by removing the extension and replacing underscores/hyphens with spaces.

The first version should not perform fuzzy geocoding, external gazetteer lookup, or database storage. Rely only on the LLM and conservative normalization.

Include a `--dry-run` option that extracts and chunks files, writes processing information, and prints summary counts without calling any LLM API.

## Pipeline

Implement the pipeline as follows.

### 1. Extract text

For EPUB:

* Use `ebooklib` and `BeautifulSoup`.
* Extract document sections in reading order if possible.
* Preserve section/chapter titles when available.
* For `source_ref`, prefer a chapter/section title when confidently available.
* If no title is available, use the EPUB item href or spine index.
* Stable provenance is more important than perfect chapter detection.

For PDF:

* Use `pymupdf`.
* Extract page text page by page.
* Preserve page numbers.
* Do not use OCR in the first version.

### 2. Chunk text

Chunk extracted text into manageable units.

Recommended:

* 6,000–10,000 characters per chunk
* overlap of 500–1,000 characters
* prefer splitting on paragraph or sentence boundaries

Each chunk should retain:

```text
work
source_file
source_type
source_ref
chunk_id
text
```

### 3. Extract places with LLM

Use the [`llm`](https://llm.datasette.io/en/stable/) Python library as the model abstraction layer instead of calling the OpenAI SDK directly.

The `--model` argument should accept an `llm` model identifier. This keeps the implementation provider-agnostic and allows OpenAI or other providers to be configured through `llm` and its plugins.

Default to a cheap, reliable, JSON-capable OpenAI model that is broadly accessible through normal OpenAI accounts. Use `gpt-4o-mini` as the default `llm` model identifier unless the installed `llm` configuration requires a different equivalent identifier. Keep the default model in one easy-to-change constant. Users can override it with `--model` for higher-quality or provider-specific runs.

Require JSON output from the model using JSON prompting and/or any structured-output features exposed by `llm` when available. Parse and validate the returned JSON defensively. Never write malformed LLM responses directly to CSV; failed parsing should be logged as a chunk failure.

For each chunk, ask the model to extract real-world place mentions only.

The model must return JSON matching this schema:

```json
{
  "mentions": [
    {
      "place_text": "string",
      "normalized_place": "string",
      "place_type": "city | street | district | landmark | building | country | region | river | natural_feature | other | null",
      "city": "string or null",
      "region": "string or null",
      "country": "string or null",
      "confidence": "number",
      "context": "string"
    }
  ]
}
```

Rules for extraction:

* Extract only places explicitly present in the text.
* Include cities, countries, regions, streets, districts, landmarks, buildings, rivers, seas, mountains, and other real geographic places.
* Include literary references to real places.
* Exclude people unless the name clearly refers to a place in context.
* Exclude purely fictional places unless the text strongly implies they correspond to a real-world place.
* Do not invent locations.
* Preserve exact mention in `place_text`.
* Normalize spelling in `normalized_place`.
* Fill `city`, `region`, and `country` only when strongly supported by the text or very common knowledge.
* Use `null` when uncertain.
* Include a short context sentence or phrase.
* `confidence` should reflect extraction and normalization certainty, not the importance of the place.
* Return an empty `mentions` array if no places are found.

### 4. Deduplicate

Deduplicate mentions conservatively.

For `place_mentions.csv`, remove exact duplicate rows caused by chunk overlap.

Keep `chunk_id` in the raw intermediate data for provenance, but deduplicate the final `place_mentions.csv` conservatively.

Suggested duplicate key:

```text
work + source_ref + place_text + normalized_place + context
```

If overlap causes near-duplicates with slightly different context, do not attempt fuzzy matching in v1. Prefer exact duplicate removal only.

For `place_index.csv`, group by:

```text
normalized_place + city + region + country
```

When conflicting normalization occurs, preserve the most frequent form and keep lower-confidence variants in the mentions table.

### 5. Error handling

The script must:

* continue processing if one file or chunk fails
* log failures in `processing_log.csv`
* retry API calls with exponential backoff
* save intermediate results periodically
* avoid reprocessing chunks already completed if rerun

### 9. Code quality

The implementation should be clear and boring.

Use:

* small functions
* type hints
* docstrings where useful
* no unnecessary frameworks
* no database in v1
* CSV output only in v1

Prefer a complete, working v1 over architectural abstraction. Do not add tests, databases, web interfaces, async queues, geocoding, or visualization code unless explicitly requested.

### 10. Acceptance criteria

The tool is complete when:

* dependencies are managed with `uv` using `pyproject.toml` and `uv.lock`
* it can be installed with `uv sync`
* it can be run with `uv run python -m src.place_index --input data --output output`
* it scans all PDFs and EPUBs in `data/`
* it produces the three CSV outputs
* failed chunks are logged, not fatal
* output rows include provenance back to work and page/chunk
* intermediate chunk results are checkpointed in `output/intermediate_mentions.jsonl`
* rerunning with `--resume` does not duplicate existing results
* `--dry-run` works without an API key and does not call any LLM API
* the code is easy to modify for future geocoding
